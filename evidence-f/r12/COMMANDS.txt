WHAT R12 WAS (STAGE-E.md §9.12): V2's catch-binder taint over-approximated. The old
trigger `throwsNonLiteral` tainted the catch binder whenever the try block threw
ANYTHING that was not a bare literal atom (KNum/KStr/KBool/…). Two defects fell out:
  (a) FALSE REJECTION: a throw of a composite built only from literals — a concat
      ("err-" + "42") or a template — carries no vault value, yet the catch binder
      was tainted and a boundary sink over it was refused (safe code rejected).
  (b) LATENT ESCAPE: `throwsNonLiteral` leaned on isLiteralNode, which treated EVERY
      template as a literal — so `throw `+"`err ${owner}`"+` produced a CLEAN binder and
      write(e) ADMITTED a real pii leak.

THE FIX (internal/admission/flow.go): replaced `throwsNonLiteral` with
`throwsPossiblyPii`, driven by a new `provablyCleanThrow(expr)` predicate. A throw is
provably clean iff its AST contains ZERO reference forms (no KLocal/KRef/KSelfRef/
KName/KCall/KMember/KIndex) anywhere — literals, and literal-composites (KBinary/
KUnary/KCond/KTypeof/KArray/KObject/KTemplate over clean parts). Vault values enter an
expression ONLY through those reference forms, so a reference-free throw cannot carry
pii in ANY scope — no env/De-Bruijn resolution needed. Fail-closed: any reference (or
unrecognized node, or object spread / computed key / array spread) is unclean.

SOUNDNESS: provablyCleanThrow is a sound over-approximation of "clean" — it may call a
genuinely-clean throw unclean (binder stays conservatively tainted), but NEVER calls a
possibly-pii throw clean. No real pii escape is admitted.
